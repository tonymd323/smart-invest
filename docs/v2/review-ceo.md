# JARVIS 投资系统 2.0 — CEO 评审报告

_评审人：JARVIS（CEO 视角）_  
_日期：2026-03-27_  
_评审范围：PRD / Architecture / Progress / Dev-Plan_

---

## 一、战略完整性

### 1. [严重] 投资决策链路缺少「卖出决策」环节

PRD 覆盖了「发现 → 买入」的完整链路，但**卖出端几乎空白**。超预期/扣非新高是「持有信号」，回调买入是「买入信号」，但什么时候卖？止盈策略呢？止损策略呢？

> **建议：** P1 中至少加入「止盈/止损信号检测」模块。哪怕 MVP 只是"涨幅超 20% 提醒 Tony 考虑减仓"，也比完全缺失强。

### 2. [严重] 缺少「仓位管理」维度

系统只告诉你"买什么"，不告诉你"买多少"。Tony 是个人投资者，仓位控制是风控的核心。

> **建议：** 至少在 Bitable 同步中增加「建议仓位比例」字段，哪怕初期只是简单的等权分配。

### 3. [建议] 缺少「市场情绪/大盘趋势」宏观判断

当前系统完全基于个股维度分析，没有大盘/板块级别的信号。在熊市里回调买入可能是接飞刀。

> **建议：** P1 中考虑加入「大盘趋势判断」作为买入信号的过滤条件（如沪深300均线位置）。

### 4. [严重] 「发现池」的自动发现逻辑未定义

PRD 提到发现池是"自动"的，但**如何发现？依据什么？** 从 5000 只 A 股中怎么筛选进入发现池？这个核心逻辑在 PRD 和 Architecture 中都没有明确。

> **建议：** 明确发现池的入场规则（如：最近一季净利润增速>30% 且 PE<30），这是产品核心差异化之一。

## 二、需求优先级

### 5. [严重] P-06 数据校验被标为 P1，应为 P0

没有数据校验的金融系统是定时炸弹。Pipeline 写入脏数据 → Analyzer 输出错误信号 → Tony 做出错误决策 → 真金白银损失。

> **建议：** P-06 升级为 P0。至少需要：数值范围校验、缺失值检测、数据一致性检查（如利润表 vs 资产负债表勾稽）。

### 6. [建议] I-04 错误处理被标为 P1，应为 P0

生产环境中任何 Provider 超时/返回异常数据都需要优雅处理。现在架构里写了"自动降级"，但错误处理框架是 P1，逻辑矛盾。

> **建议：** 错误处理至少升级为 P0 的最小集（异常捕获 + 日志 + 降级触发），高级功能（重试队列、告警）可为 P1。

### 7. [建议] A-07 回测标为 P1 合理，但缺少验证框架

没有回测，所有的买入/卖出策略都是"直觉"。建议在 M6 集成测试中加入「输出结果对比 1.0」的强制步骤，先把历史数据验证做起来。

### 8. [表扬] P0 划分整体合理

核心链路（采集→分析→推送）的 16 项 P0 覆盖了端到端 MVP，三件套齐了。

## 三、用户价值

### 9. [表扬] 「事件驱动实时推送」是最高价值功能

Tony 是上班族，不可能盯盘。系统替他盯着，有事才打扰，没事不废话 —— 这是真正的私人助理价值。这是 2.0 最核心的差异化。

### 10. [严重] 日报推送时间设计有 UX 问题

跟踪池日报 15:30（收盘后 15 分钟）、发现池日报 21:00。

- 15:30 Tony 可能还在上班或路上，信息过载
- 21:00 发现池日报和 Bitable 同步内容高度重复

> **建议：** 
> - 跟踪池日报改为 18:00（下班后，有时间消化）
> - 发现池日报和跟踪池日报合并，或明确区分两份报告的差异价值

### 11. [建议] 缺少「静默模式」或「免打扰」设计

如果新闻事件检测每 30 分钟跑一次，一天可能推 10+ 条 DM。Tony 在开会/开车/陪女儿时被打扰怎么办？

> **建议：** 增加消息聚合机制（如"30 分钟内最多推 1 条"）或基于日历判断忙闲状态。

### 12. [建议] ROI 最高的 3 个功能排序

对 Tony 来说：
1. **事件驱动推送**（省时间、救命） — ROI 最高
2. **回调买入评分**（直接帮赚钱） — ROI 高
3. **超预期/扣非新高扫描**（发现机会） — ROI 中高
4. 日报（全景监控） — ROI 中等
5. Bitable 同步（数据沉淀） — ROI 较低但必需

## 四、风险识别

### 13. [严重] 最大风险：东方财富 API 的稳定性

架构中 5 个 Provider 有 4 个依赖东方财富。"自动降级"写了但没经过压力测试。如果东财 API 在关键时刻（如财报季）挂了，整个系统瘫痪。

> **建议：** 
> - 增加「降级测试」到 M6（dev-plan 已提到，但需确认要测的场景）
> - 考虑加入「爬虫兜底」方案（scrapling-crawler skill 已有）
> - 关键数据源（行情）至少有 3 个备选

### 14. [严重] SQLite 并发写入风险

Pipeline 串行写入 + 盘中每 30 分钟新闻采集写入 + Bitable 同步 —— 多个进程/线程同时读写 SQLite。SQLite 的写锁机制会导致 `database is locked` 错误。

> **建议：** 
> - 明确写入策略：Pipeline 期间暂停新闻采集，或使用 WAL 模式 + 合理的 retry
> - 架构文档中应标注 SQLite 的并发限制和应对方案

### 15. [严重] 工时估算过于乐观

总计 9.5h（含 1h 验收）。考虑到：
- 5 个 Provider 需要对接不同的 API（各有坑）
- 事件检测算法需要调参
- 降级逻辑需要模拟测试
- 灰度切换需要和 1.0 并行

> **建议：** 乘以 1.5-2 倍，即 **14-19 小时**。尤其 Phase 1 的 Provider 抽象层，2 小时大概率不够（API 调研 + 异常处理 + 测试）。

### 16. [建议] 单点故障：小马一个人扛所有开发

Phase 1-3 全部依赖小马。如果小马遇到困难或时间冲突，整个项目卡住。

> **建议：** 至少把 Phase 1 拆成可验证的小任务，JARVIS 或小巴可以做代码审查和测试辅助。

### 17. [建议] 缺少「数据安全」考量

stocks.json 存储持仓信息、SQLite 存储财务数据 —— 这些是 Tony 的投资隐私。

> **建议：** 明确数据存储位置（本地 vs 云端）、访问控制（谁能读）、备份策略。至少在非功能需求中加一条。

## 五、竞争分析

### 18. [表扬] 核心差异化清晰

| 功能 | 同花顺/雪球 | 量化系统 | JARVIS 2.0 |
|------|------------|---------|------------|
| 财务数据 | ✅ | ✅ | ✅ |
| 超预期扫描 | ❌ 手动看 | ✅ 但门槛高 | ✅ 自动 |
| 回调买入评分 | ❌ | ✅ | ✅ |
| **事件驱动推送** | ❌ 付费且不精准 | ❌ 大多是定时跑 | ✅ **核心差异** |
| **AI 自然语言交互** | ❌ | ❌ | ✅ **核心差异** |
| 个人化定制 | ❌ | ❌ | ✅ stocks.json |

**核心护城河：** AI 自然语言 + 个性化监控 + 实时事件 = Tony 独有的私人投研助理。

### 19. [建议] 差异化可以更锐利

当前 PRD 没有把「AI 对话」作为显式功能写进去。Tony 可以问"最近有哪些超预期的股票"、"回调买入的机会怎么样"—— 这才是 JARVIS 的独有优势。

> **建议：** 在 P1 中加入「AI 问答接口」，让 Tony 可以用自然语言查询分析结果。

### 20. [建议] 缺少「学习闭环」

系统每天产生大量数据和决策，但没有记录「Tony 是否采纳了建议」以及「事后收益如何」。没有反馈就没有进化。

> **建议：** 在 Bitable 中增加「建议采纳率」和「建议收益率」跟踪，这是长期竞争力的关键。

## 六、架构评审

### 21. [表扬] 三层分离设计干净

采集 → 分析 → 推送，职责清晰，可独立测试。双池分离的思路也很好。

### 22. [建议] Cron 时间线有冲突

- 15:15 Pipeline → Analyzer → Pusher（发现池日报 + Bitable）
- 15:30 跟踪池日报
- 21:00 发现池日报

15 分钟完成 Pipeline + Analyzer + 生成日报，时间非常紧张。如果任何一个 Provider 超时，15:30 的跟踪池日报就赶不上。

> **建议：** 跟踪池日报推迟到 16:00，留出 45 分钟的 buffer。或者把跟踪池日报和发现池日报合并。

### 23. [建议] 缺少「Pipeline 超时保护」

架构描述中 Pipeline 串行调 5 个 Provider。如果某个 Provider 超时（比如 Tushare 限流），后面的全堵住。

> **建议：** 每个 Provider 设置独立超时（如 30 秒），超时自动降级或跳过，保证整体 Pipeline 在合理时间内完成。

## 七、Dev-Plan 评审

### 24. [表扬] 多 Agent 协作流程清晰

JARVIS → 小马 → 小达 → JARVIS 的审查链条合理。代码审查分级（自审/产品审/CEO审）也匹配了变更风险。

### 25. [严重] Phase 2 并行拆分有隐患

Dev-Plan 建议 Phase 2 把 Pipeline 和 Analyzer 拆成两条线并行。但 Analyzer 依赖 Pipeline 写入的数据结构（表名、字段名）。如果小马在 Pipeline 端改了 schema，Analyzer 端代码会挂。

> **建议：** Phase 2 并行的前提是先用 30 分钟对齐 SQLite schema（共识文档），然后两人各干各的。或者就老老实实串行，Phase 2 多花 1 小时，但零风险。

### 26. [建议] 缺少「验收标准」的量化定义

Dev-Plan 中 Phase 4 验收只写了"对比 1.0 和 2.0 输出结果"。但什么算通过？数据一致率 100%？95%？允许差异的边界在哪？

> **建议：** 明确验收标准：数据一致性 ≥ 99%、所有 P0 功能可通过手动触发验证、降级场景可通过测试。

## 总结

| 维度 | 评分 | 关键问题 |
|------|------|---------|
| 战略完整性 | ⚠️ 6/10 | 缺卖出决策、仓位管理、发现池入场规则 |
| 需求优先级 | ✅ 7/10 | 数据校验和错误处理应升级为 P0 |
| 用户价值 | ✅ 8/10 | 事件推送价值最高，但 UX 细节需打磨 |
| 风险识别 | ⚠️ 5/10 | 东财依赖、SQLite 并发、工时乐观 |
| 竞争分析 | ✅ 8/10 | 差异化清晰，但可以更锐利 |
| 架构设计 | ✅ 7/10 | 三层分离干净，但并发和超时需补方案 |
| 开发方案 | ✅ 7/10 | 流程清晰，但并行拆分有隐患 |

**CEO 判决：** 设计方向正确，核心思路清晰。但有 3 个严重问题必须在开工前解决：
1. **发现池入场规则** — 没有这个，发现池就是个空壳
2. **P-06 数据校验升级为 P0** — 金融数据不能有脏数据
3. **工时乘以 1.5 倍** — 别给团队制造不必要的压力

> 其余问题可以在开发过程中迭代解决。开工前先开个 30 分钟的 kickoff，对齐 schema 和发现池规则。
